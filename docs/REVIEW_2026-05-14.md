## Critical (will corrupt results or crash a run)

### #1 ‚Äî N_BUILDINGS mismatch in nb 05
**Where:** `notebooks/05_sft_gemma_colab.ipynb`, cells 15 (SFT setup) and 24 (eval env factory).

**What was wrong:** `N_BUILDINGS = 6` while `dump_sac_trajectory_jsonl` in nb 04 was called with `building_slices=[[0,1,2],[3,4,5]]` and the per-row width defaulted to the slice width (3). Every training example had: system prompt "manage 6 buildings", state showing 3 buildings, response with 3 action lines. The eval cells then built `make_colab_env(...)` whose default `buildings=BUILDINGS=[0..5]` (6 buildings) ‚Äî OOD vs training.

**Fix:** `N_BUILDINGS = 3`; `make_colab_env` defaults to `TRAINING_BUILDINGS=[0,1,2]`; switched to `src.env.make_env` so the schema source matches nb 04/05 (same dataset file).

### #2 ‚Äî `NameError: _ACTION_RE` in nb 06 SFTProvider
**Where:** `notebooks/06_eval_generalization.ipynb`, cell 11.

**What was wrong:** Cell imported `_ACTION_TAG_RE` (opening-tag regex) but its `step()` method called `_ACTION_RE.search(last)` (full-match regex), which is never defined. Same bug as commit `e543a387` fixed in nb 05 ‚Äî not propagated to nb 06.

**Fix:** Changed the call to `_ACTION_TAG_RE.search(last)`.

### #3 ‚Äî `sys_p_cot` referenced but never assigned in nb 05
**Where:** `notebooks/05_sft_gemma_colab.ipynb`, cells 44 and 46.

**What was wrong:** Both cells use `sys_p_cot` but it's never bound. The historical "CoT eval blowup" referenced in PROGRESS.md 2026-05-12 was the symptom of this `NameError`.

**Fix:** User-handled ‚Äî user planned to delete cells 13+ in nb 05 themselves.

### #4 ‚Äî (obsolete)
This item concerned a divergence in the project's secondary reward function between nb 01 and `src/env.py`. That reward function was removed on 2026-05-17 ‚Äî `MERLINReward` is now the only reward ‚Äî so this item no longer applies.

### #5 ‚Äî Prompt buckets didn't match the bucket functions
**Where:** `src/agent.py:make_minimal_prompt`, `src/sft.py:make_sft_prompt`.

**What was wrong:** Both prompts told the model "'price' (LOW / MID / PEAK)" and "'solar' (NONE / LOW / MID / HIGH)", but the bucket functions only emit two price levels (`LOW`, `PEAK`) and three solar levels (`NONE`, `LOW`, `HIGH`). The model was told categories existed that it would never observe.

**Fix:** Prompt strings now read `(LOW / PEAK)` for price and `(NONE / LOW / HIGH)` for solar ‚Äî matching what the buckets actually emit. Carbon (`LOW / MID / HIGH`) already matched.

### #6 ‚Äî Silent LoRA warm-start failure in nb 07
**Where:** `notebooks/07_grpo_colab.ipynb`, cell 13.

**What was wrong:** The original code called `model.load_adapter(SFT_ADAPTER_PATH, adapter_name="default", is_trainable=True)` inside a `try/except`. `get_peft_model` had already attached a fresh "default" adapter, so `load_adapter` with the same name typically errors. The `except` clause swallowed the error and continued silently with the freshly-initialised LoRA ‚Äî "RL from SFT" degraded to "RL from base" with only a printed warning.

**Fix:** Now uses `peft.set_peft_model_state_dict(model, sft_sd, adapter_name="default")` to stamp the SFT weights directly into the existing default adapter. After the stamp, asserts that a probe LoRA-A weight changed ‚Äî hard-fails if the warm-start didn't take, so silent degradation can't happen.

### #7 ‚Äî KL term could go negative in nb 07 GRPO loss
**Where:** `notebooks/07_grpo_colab.ipynb`, cell 21.

**What was wrong:** `kl_terms.append((lp_pol - lp_ref).mean())` ‚Äî this is an unbiased estimator of `KL(œÄ_pol ‚Äñ œÄ_ref)` only in expectation under samples from `œÄ_pol`. For any finite sample it can go negative, in which case the optimiser is **rewarded** for moving away from the reference policy.

**Fix:** Replaced with the k3 estimator (Schulman):
```python
log_ratio  = (lp_pol - lp_ref).clamp(min=-10.0, max=10.0)
kl_per_tok = log_ratio.exp() - 1.0 - log_ratio   # element-wise ‚â• 0
kl_terms.append(kl_per_tok.mean())
```
This is what TRL / DeepSpeed use in production GRPO/PPO loops. Clamp prevents `exp()` overflow if the policy drifts far early in training.

### #8 ‚Äî RL prompt was OOD vs SFT in nb 07
**Where:** `notebooks/07_grpo_colab.ipynb`, cell 17.

**What was wrong:** Cell imported and used `make_minimal_prompt` (the CoT prompt with `<thought>` block). The SFT'd adapter was trained on `make_sft_prompt` (no-CoT). At RL start, the SFT'd model would have produced garbage `<thought>` tokens (blowing the generation budget) and emitted actions inside the thought block (invisible to `parse_actions`).

**Fix:** Replaced with `make_sft_prompt(N_BUILDINGS)`. This is exactly the issue PROGRESS.md 2026-05-12 calls out for nb 05 ¬ß 19.

## High (wrong values, but the run will complete)

### #9 ‚Äî Off-by-one in nb 07 `rollout_window`
`make_colab_env(start=t0, end=t0 + WINDOW_STEPS, ...)` provisions `WINDOW_STEPS + 1` env steps (CityLearn's `simulation_end_time_step` is inclusive) but the loop runs only `WINDOW_STEPS`. **Fix:** `end = t0 + WINDOW_STEPS - 1`.

### #10 ‚Äî `action_to_token` rounding bias
`pct = int(round(abs(a) * 100 / 20) * 20)` used Python 3's banker's rounding. Empirical bucket widths:

| Bucket    | `|a|` range  | Width |
|-----------|--------------|-------|
| CHARGE_20 | [0.10, 0.30) | 0.20  |
| CHARGE_40 | [0.30, 0.51) | 0.21  |
| CHARGE_60 | [0.51, 0.70) | **0.19** (squeezed) |
| CHARGE_80 | [0.70, 0.91) | 0.21  |
| CHARGE_100| [0.91, 1.00] | **0.09** (squeezed) |

`CHARGE_60` and `CHARGE_100` got less SAC mass than they should. **Fix:** integer round-half-up:
```python
units = int(round(abs(a) * 100))      # 0..100
pct   = ((units + 10) // 20) * 20     # 0.30‚Üí40, 0.50‚Üí60, 0.70‚Üí80, 0.90‚Üí100
```
Verified post-fix: uniform 0.20-wide buckets, symmetric for charge / discharge.

### #16 ‚Äî SAC training on stale obs vector ‚ùå RETRACTED

**Original claim:** the obs vector returned by `env.step()` always reports `electrical_storage_soc = 0` and `net_electricity_consumption = 0`, so the bundled CityLearn SAC sees zero SoC throughout training. Confirmed empirically in my test, then verified in CityLearn **2.5.0** source.

**Retraction:** the project pins CityLearn **2.6.0b2** in every install cell. 2.6.0b2 introduced
```python
endogenous_t = t if include_all else max(t - 1, 0)
'electrical_storage_soc': building.electrical_storage.soc[endogenous_t],
```
in `BuildingOpsService.get_observations_data()`. For `include_all=False` (the agent-facing call) it reads `soc[t-1]` ‚Äî the just-realised value. The bug is fixed in the version the project actually uses.

My test ran on local `python3` which has CityLearn 2.5.0 installed system-wide, *not* the 2.6.0b2 of the Colab pipeline. SAC training in nb 04 was correct all along.

[docs/CITYLEARN_INSIGHTS.md](docs/CITYLEARN_INSIGHTS.md) ¬ß 1 now notes this is a 2.5-only bug.
